## Раздел 1: Описание проблемы

### Что такое Race Condition?

Race condition (состояние гонки) - это ситуация, когда несколько процессов одновременно обращаются к одним и тем же данным в базе данных и пытаются их изменить. Результат выполнения зависит от того, в каком порядке и с какой скоростью выполняются операции.

**Пример из жизни:**
- Два человека одновременно покупают последний билет на поезд на сайте. Для обоих билет еще доступен, и они нажимают «Купить». Если система не контролирует одновременный доступ к данным, оба запроса могут обработаться успешно. Билет будет продан два раза, хотя фактически он был один.

### Почему READ COMMITTED не защищает от двойной оплаты?

На уровне изоляции READ COMMITTED:
1. Каждая транзакция видит только те данные, которые были закоммичены другими транзакциями на момент выполнения каждого отдельного оператора SELECT.
2. Когда две транзакции одновременно выполняют SELECT, обе видят `status = 'created'`(ни одна из них еще не выполнила COMMIT).
3. В результате обе выполняют UPDATE и INSERT в историю.

**Демонстрация проблемы:**

```
Время | Сессия 1                    | Сессия 2
------|----------------------------|---------------------------
t1    | BEGIN                      |
t2    | SELECT status (created)    |
t3    |                            | BEGIN
t4    |                            | SELECT status (created)
t5    | pg_sleep(3)                |
t6    |                            | pg_sleep(3)
t7    | UPDATE status = paid       |
t8    |                            | UPDATE status = paid (ждет)
t9    | COMMIT                     |
t10   |                            | UPDATE выполняется...
t11   |                            | COMMIT
```

Что происходит на каждом этапе:
- **t1–t2:** Сессия 1 начинает транзакцию и читает статус заказа - получает `created`.
- **t3–t4:** Сессия 2 начинает свою транзакцию и тоже читает `created`, потому что Сессия 1 еще ничего не изменила.
- **t5–t6:** Обе сессии выполняют задержку (имитация обработки некоторых операций).
- **t7:** Сессия 1 выполняет UPDATE и блокировка строки активируется, но транзакция еще не завершена.
- **t8:** Сессия 2 пытается UPDATE ту же строку, но блокировка занята - она ждет.
- **t9:** Сессия 1 делает COMMIT, снимая блокировку.
- **t10:** Сессия 2 получает доступ и выполняет UPDATE. READ COMMITTED не проверяет не перечитывает данные и выполняет запись `paid` в историю.
- **t11:** Сессия 2 делает COMMIT. В таблице  теперь 2 записи `paid`.

### Примеры из реальной жизни

1. **Двойной клик на кнопку "Оплатить"**
   - Пользователь нажимает кнопку дважды
   - Два HTTP-запроса приходят почти одновременно
   - Оба запроса начинают обработку параллельно

2. **Микросервисная архитектура**
   - Пользователь оформляет заказ в интернет-магазине
   - Платежный сервис отправляет событие в сервис заказов о том, что оплата прошла.
   - Из-за сбоя сети сообщение отправляется повторно.
   - Два экземпляра сервиса заказов получают одно и то же сообщение и оба создают заказ в системе.
   - В результате в системе появляется два одинаковых заказа.

3. **Сбой сети**
   - Клиент отправляет запрос на оплату
   - Ответ от сервера теряется из-за разрыва соединения
   - Клиент не знает, прошла ли оплата, и повторяет запрос
   - Первый запрос уже выполняется на сервере. Второй запрос запускается параллельно с первым

## Раздел 2: Уровни изоляции в PostgreSQL

### READ UNCOMMITTED

**Описание:**
Самый слабый уровень изоляции. Может читать незакоммиченные изменения других транзакций.

**Предотвращает:**
- Ничего.

**Не предотвращает:**
- Dirty reads (по стандарту SQL; в PostgreSQL — предотвращает)
- Non-repeatable reads
- Phantom reads

**Когда использовать:**
Почти нет смысла использовать этот уровень. Может применяться для аналитических запросов, где допустимы приблизительные результаты ради максимальной скорости.

**Особенность в PostgreSQL:**
В PostgreSQL READ UNCOMMITTED работает как READ COMMITTED из-за архитектуры MVCC.

---

### READ COMMITTED (по умолчанию)

**Описание:**
READ COMMITTED - это уровень изоляции транзакций, при котором запросы могут читать только зафиксированные (committed) данные.

Каждый SELECT в транзакции видит только те данные, которые были закоммичены до начала выполнения запроса.

**Предотвращает:**
- ✅ Dirty reads (чтение незакоммиченных данных)

**Не предотвращает:**
- ❌ Non-repeatable reads (повторное чтение дает другой результат)
- ❌ Phantom reads (новые строки появляются в результате запроса)

**Пример non-repeatable read:**
```sql
-- Сессия 1
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- Результат: 1000
-- Сессия 2 изменяет balance и делает COMMIT
SELECT balance FROM accounts WHERE id = 1; -- Результат: 500 (!)
COMMIT;
```

**Когда использовать:**
Подходит для большинства повседневных операций: чтение каталога товаров; поиск, чтение информации о пользователе. Подходит, где допустимо изменение данных между запросами в одной транзакции

---

### REPEATABLE READ

**Описание:**
Транзакция видит snapshot данных, зафиксированный на момент выполнения первого оператора в транзакции. Все последующие SELECT внутри этой транзакции будут возвращать одни и те же данные, даже если другие транзакции тем временем изменяли и коммитили строки.

Транзакция видит snapshot данных на момент начала и работает с ним до самого COMMIT.

**Предотвращает:**
- ✅ Dirty reads
- ✅ Non-repeatable reads
- ✅ Phantom reads (в PostgreSQL благодаря MVCC)

**Не предотвращает:**
- ❌ Write skew (специфичная аномалия)
- ❌ Некоторые сериализационные аномалии

**Особенность в PostgreSQL:**
В отличие от стандарта SQL, PostgreSQL на REPEATABLE READ также предотвращает phantom reads.

**Когда использовать:**
Используется в ситуациях, когда нужно, чтобы в рамках одной транзакции все чтения возвращали одинаковые данные: формирование отчетов, финансовые операции (оплата, перевод средств), резервирование товаров на складе.

---

### SERIALIZABLE

**Описание:**
Самый строгий уровень изоляции, гарантирующий, что результат параллельного выполнения транзакций будет таким, как если бы транзакции выполнялись последовательно.

**Предотвращает:**
- ✅ Все аномалии чтения
- ✅ Сериализационные аномалии
- ✅ Write skew

**Недостатки:**
- ❌ Может откатывать транзакции при конфликтах (serialization failure)
- ❌ Снижение производительности
- ❌ Требует retry logic в приложении

**Пример serialization failure:**
```sql
-- Сессия 1
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 100 WHERE id = 1;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 200 WHERE id = 2;

-- При COMMIT одна из транзакций получит ошибку:
-- ERROR: could not serialize access due to read/write dependencies
```

**Когда использовать:**
Применяется в системах, где абсолютная корректность важнее производительности: банковские переводы между счетами, системы бронирования (билеты на поезд/самолет).

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ решает проблему?

REPEATABLE READ использует snapshot isolation:
1. Транзакция видит снимок данных, зафиксированный на момент начала транзакции.
2. Это означает, что вторая транзакция не увидит изменений, даже если первая уже закоммитилась
3. Однако, без блокировок (FOR UPDATE) обе транзакции могут одновременно прочитать `status = 'created'` из своих снимков. 
4. Первая транзакция успешно выполнит UPDATE. Вторая получит ошибку сериализации на этапе UPDATE.

### Зачем нужен FOR UPDATE?

FOR UPDATE гарантирует, что только одна транзакция сможет прочитать и изменить строку в данный момент времени.
С FOR UPDATE первая транзакция блокирует строку на этапе SELECT. Вторая транзакция останавливается на SELECT и ждет, пока первая завершится.

`FOR UPDATE` создает эксклюзивную блокировку на уровне строки (row-level lock):
 

**Без FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...';
-- Другая транзакция может прочитать ту же строку
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**С FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...' FOR UPDATE;
-- Другая транзакция ЖДЕТ освобождения блокировки
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**Типы блокировок:**
- `FOR UPDATE` — эксклюзивная блокировка (никто не может изменить или заблокировать)
- `FOR SHARE` — разделяемая блокировка (другие могут читать с FOR SHARE, но не могут изменять)
- `FOR NO KEY UPDATE` — как FOR UPDATE, но разрешает concurrent FOR KEY SHARE
- `FOR KEY SHARE` — слабая блокировка (предотвращает DELETE и UPDATE ключевых полей)

### Что произойдет без FOR UPDATE на REPEATABLE READ?

Даже на REPEATABLE READ возможна аномалия:

```sql
-- Сессия 1
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
COMMIT;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created (snapshot!)
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
-- ЧТО ПРОИЗОЙДЕТ?
```

**Что произойдёт:**

Сессия 1 успешно выполнит UPDATE и COMMIT. Когда Сессия 2 попытается выполнить UPDATE, PostgreSQL обнаружит, что строка была изменена транзакцией, закоммиченной после начала снимка Сессии 2. На уровне REPEATABLE READ это приводит к ошибке:could not serialize access due to concurrent update

Сессия 2 будет откачена. Это защищает от двойной оплаты, но ошибка возникает поздно, код приложения уже мог выполнить побочные эффекты (например отправить email).

### Разница между FOR UPDATE и FOR SHARE

| Характеристика | FOR UPDATE | FOR SHARE |
|----------------|------------|-----------|
| Тип блокировки | Эксклюзивная | Разделяемая |
| Блокирует чтение с FOR UPDATE | ✅ | ✅ |
| Блокирует чтение с FOR SHARE | ✅ | ❌ |
| Блокирует UPDATE | ✅ | ✅ |
| Блокирует DELETE | ✅ | ✅ |
| Use case | Перед изменением | Защита от изменений |

**Пример использования FOR SHARE:**
```sql
-- Проверка наличия товара перед созданием заказа
SELECT quantity FROM products WHERE id = '...' FOR SHARE;
-- Товар не может быть удален или изменен до COMMIT
```

## Раздел 4: Рекомендации для продакшена

### Какой ISOLATION LEVEL использовать для продакшена маркетплейса?

**Рекомендация:** Для продакшена маркетплейса рекомендуется использовать **READ COMMITTED как default уровень** с явным использованием **REPEATABLE READ + FOR UPDATE для критичных операций**.

#### Обоснование:

**1. Производительность**
- READ COMMITTED имеет минимальный overhead - не требует отслеживания фантомных записей, что экономит память и ускоряет работу.
- REPEATABLE READ используется только там, где необходимо - для платежных операций, резервирования товаров и изменения баланса (обычно это менее 5% запросов).
- Избегаем глобального использования SERIALIZABLE, который может привести к частым rollback-ам из-за конфликтов.

**2. Безопасность данных**
- Критичные операции (оплата, изменение баланса) защищены через REPEATABLE READ + FOR UPDATE - гарантировано другая транзакция не изменит данные. 
- Некритичные операции (просмотр каталога) работают на READ COMMITTED так как допускаемые аномалии (non-repeatable reads) не влияют на корректность логики.
- Четкое разделение зон ответственности: разработчик явно указывает нужные уровни изоляции только для тех методов, где это необходимо.

**3. Риски deadlock**
- FOR UPDATE может привести к deadlock при неправильном порядке блокировок. Например, Сессия 1 блокирует заказ A, затем пытается блокировать заказ B, а Сессия 2 наоборот.
- Решение: всегда блокировать ресурсы в одном порядке, например, по возрастанию UUID или ID.
- Пример: сначала order, потом order_items, потом payment. Если нужно блокировать несколько заказов, то сортировать по ID и блокировать последовательно.

**4. Простота разработки**
- READ COMMITTED легко понять и отлаживать. Каждый SELECT видит актуальные данные.
- REPEATABLE READ + FOR UPDATE требует explicit intent - разработчик явно показывает намерение изменить данные.
- Код становится самодокументируемым, так как наличие FOR UPDATE показывает, что здесь обрабатывается критическая секция.

**5. Масштабируемость**
- READ COMMITTED хорошо масштабируется потому что он создает меньше блокировок и конфликтов между транзакциями.
- Локальное использование блокировок не создает bottleneck: FOR UPDATE блокирует только одну строку заказа, не мешая параллельной оплате других заказов.
- Connection pooling работает эффективно — транзакции короткие, соединения быстро возвращаются в пул. При использовании SERIALIZABLE транзакции могут откатываться и перезапускаться, увеличивая время занятости соединения.

### Альтернативные подходы

#### Подход 1: Использовать SERIALIZABLE везде

**Плюсы:**
- Максимальная корректность
- Не нужно думать о блокировках
- PostgreSQL автоматически обнаруживает конфликты

**Минусы:**
- Значительное снижение производительности (20-50%)
- Требует retry logic во всех операциях
- Высокий процент rollback при нагрузке
- Сложная отладка serialization failures

**Вывод:** Не рекомендуется для high-load систем.

#### Подход 2: Optimistic Locking (версионирование)

**Реализация:**
```sql
ALTER TABLE orders ADD COLUMN version INTEGER DEFAULT 1;

UPDATE orders
SET status = 'paid', version = version + 1
WHERE id = '...' AND status = 'created' AND version = 1;

-- Проверить ROW_COUNT, если 0 — конфликт
```

**Плюсы:**
- Нет блокировок на чтение
- Хорошая производительность
- Масштабируется горизонтально

**Минусы:**
- Требует изменения схемы БД
- Требует retry logic в приложении
- Может быть много конфликтов при высокой нагрузке

**Вывод:** Хороший подход для распределенных систем.

#### Подход 3: Advisory Locks

**Реализация:**
```sql
BEGIN;
SELECT pg_advisory_xact_lock(hashtext('order_' || order_id));
-- Критическая секция
COMMIT; -- Блокировка автоматически снимается
```

**Плюсы:**
- Гибкий контроль блокировок
- Можно блокировать логические ресурсы
- Работает на любом уровне изоляции

**Минусы:**
- Легко забыть снять блокировку
- Сложнее отлаживать
- Требует дисциплины от разработчиков

**Вывод:** Полезно для специфичных случаев, но не как основной подход.

### Итоговая рекомендация

Для продакшена маркетплейса используйте **гибридный подход**:

1. **Default isolation level:** `READ COMMITTED`
   - Для обычных операций (чтение каталога, просмотр истории)
   - 95% операций

2. **REPEATABLE READ + FOR UPDATE:** для критичных операций
   - Оплата заказа
   - Изменение баланса
   - Резервирование товара
   - 5% операций

3. **Дополнительно:** Optimistic locking для операций с высокой конкурентностью
   - Обновление счетчиков
   - Изменение рейтингов

**Пример кода:**
```python
async def pay_order(order_id: UUID):
    async with db.transaction(isolation='repeatable_read'):
        # Заблокировать заказ
        order = await db.fetch_one(
            "SELECT * FROM orders WHERE id = $1 FOR UPDATE",
            order_id
        )

        if order['status'] != 'created':
            raise OrderAlreadyPaidError()

        # Обновить статус
        await db.execute(
            "UPDATE orders SET status = 'paid' WHERE id = $1",
            order_id
        )

        # Записать в историю
        await db.execute(
            "INSERT INTO order_status_history (order_id, status) VALUES ($1, 'paid')",
            order_id
        )
```

## Заключение

В ходе лабораторной работы было изучено:
1. Механизм возникновения race condition при конкурентных оплатах.
2. Уровни изоляции транзакций (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE).
3. Применение REPEATABLE READ + FOR UPDATE для предотвращения race condition.

Основные выводы:
- READ COMMITTED (уровень по умолчанию) не защищает от race condition при конкурентных модификациях, если код приложения полагается на результат SELECT для принятия решений.
- REPEATABLE READ + FOR UPDATE - решение для критичных операций. Блокировка строки обеспечивает корректность.
- В продакшене следует применять гибридный подход: READ COMMITTED для массовых операций чтения и REPEATABLE READ + FOR UPDATE для критичных операций.
